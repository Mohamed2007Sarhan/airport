# وثيقة شرح نظام إدارة المطارات (Airport Management System)

هذه الوثيقة تشرح بالتفصيل بنية النظام الهندسية، مسار تدفق البيانات، آلية منع الحجز المزدوج، وكيفية اختبار العمليات البرمجية المختلفة.

---

## 1. البنية الهندسية للنظام (System Architecture)

تم تصميم النظام باعتماد هيكلية مجزأة مرنة (Modular Architecture) باستخدام إطار العمل **NestJS**، مما يضمن سهولة التطوير والفصل الكامل للمسؤوليات (Separation of Concerns).

### طبقات النظام الرئيسية (Architectural Layers):
1. **طبقة المعالجة والعرض (GraphQL Layer)**:
   * تعتمد على أسلوب **Code-First**. يتم تعريف الكيانات (Entities) والمدخلات (DTOs) برمجياً بلغة TypeScript ومزيناتها، ويفوم النظام تلقائياً بتوليد مخطط الـ GraphQL (`schema.gql`).
   * تستقبل هذه الطبقة طلبات الاستعلام (Queries) والتعديل (Mutations) والاشتراكات الفورية (Subscriptions).
2. **طبقة الأمان وصلاحيات الوصول (Security & Guards)**:
   * **GqlAuthGuard**: يتحقق من رمز التوثيق (JWT Token) المرسل مع طلبات المستخدمين ويستخلص بيانات الحساب.
   * **RolesGuard**: يتحقق من دور المستخدم (Admin, Staff, Passenger) ويطابقه مع الصلاحيات المحددة لكل عملية برمجية عبر المزين `@Roles()`.
3. **طبقة منطق العمل (Business Logic - Services)**:
   * تمثل الدماغ المفكر للنظام؛ حيث تُنفذ فيها كافة الشروط والتحققات المطلوبة (مثل توافر المقاعد، صلاحيات تعيين الطاقم للرحلات، ومعالجة أوزان الأمتعة).
4. **طبقة قواعد البيانات والاتصال (TypeORM / PostgreSQL)**:
   * تدير حفظ وتعديل البيانات بشكل مرن ومؤمن باستخدام علاقات برمجية واضحة وجداول مبنية بكفاءة تدعم الفهرسة (Indexing) لتحسين سرعة الاستعلامات.
5. **طبقة معالجة المهام الخلفية (BullMQ / Redis)**:
   * تستخدم لمعالجة العمليات الثقيلة (مثل إرسال رسائل تأكيد الحجز وتنبيهات تأخر الرحلات) بشكل غير متزامن لتفادي تأخير استجابة المستخدم النهائي.

---

## 2. مسار مرور البيانات (Data Flow)

يوضح هذا القسم كيفية انتقال البيانات ومعالجتها عبر النظام في أهم العمليات البرمجية:

### أ. مسار حجز الرحلات (Booking Flow)
1. يرسل الراكب (Passenger) طلب حجز مقعد محدد على رحلة معينة عبر الـ Mutation `bookFlight`.
2. يتم التحقق من هوية الراكب وصلاحيته في طبقة الـ Guards.
3. تستدعي الـ `BookingResolver` الدالة المسؤولة في الـ `BookingService`.
4. تبدأ الـ Service **عملية برمجية مؤمنة (Database Transaction)** وتنفذ الخطوات التالية:
   * جلب الرحلة مع تفعيل **قفل الكتابة التشاؤمي (Pessimistic Write Lock)** لمنع تضارب البيانات.
   * التحقق من وجود مقاعد شاغرة في الرحلة.
   * التحقق من عدم حجز المقعد المطلوبة مسبقاً من راكب آخر.
   * إنشاء سجل الحجز، وتعديل عدد مقاعد الرحلة المتاحة بالخصم.
5. بعد إتمام العملية وحفظها بنجاح في قاعدة البيانات، يتم دفع مهمة إرسال إشعار التأكيد إلى طابور **BullMQ**.
6. يعود تأكيد الحجز فوراً للمستخدم.

### ب. مسار تحديث حالة الرحلة والاشتراكات (Status Updates & Subscriptions)
1. يقوم المدير (Admin) أو الموظف (Staff) بتحديث حالة الرحلة (مثلاً إلى `DELAYED`) عبر `updateFlightStatus`.
2. تقوم الـ `FlightService` بتحديث السجل في قاعدة البيانات.
3. يتم إطلاق عمليتين متزامنتين:
   * **النشر الفوري (PubSub)**: يتم إرسال تحديث فوري عبر الـ WebSocket لكافة واجهات المستخدمين المشتركين في قناة متابعة الرحلة (`flightStatusUpdated`).
   * **إضافة تنبيه للطابور الخلفي (BullMQ)**: يتم جلب قائمة كافة الركاب الذين لديهم حجوزات مؤكدة على هذه الرحلة، وإرسال تنبيه تأخير لكل منهم بشكل غير متزامن عن طريق الـ Worker في الخلفية.

### ج. مسار جلب البيانات بكفاءة (DataLoader Flow)
1. عند طلب قائمة الرحلات مع تفاصيل مطاري الإقلاع والوصول.
2. يتلقى الـ `FlightResolver` الطلب ويستدعي الـ DataLoader الخاص بالمطارات (`AirportLoader`).
3. يقوم الـ Loader بتجميع كافة معرفات (IDs) المطارات المطلوبة من كل الرحلات المعروضة في ذاكرة مؤقتة لمرة واحدة (Batching).
4. ينفذ استعلاماً واحداً فقط لقراءة بيانات جميع المطارات المطلوبة دفعة واحدة باستخدام جملة `IN` في الـ SQL، متجنباً مشكلة الـ N+1 الشائعة في الـ GraphQL.

## 3. آلية منع الحجز المزدوج (Double Booking Prevention)

تعد مشكلة الحجز المزدوج (أن يقوم مسافران بحجز نفس المقعد في نفس الوقت) من أخطر المشاكل التي تواجه أنظمة الطيران. تم حل هذه المشكلة على مستويين لضمان أمان تام:

### أولاً: مستوى قاعدة البيانات (Database Constraint)
تم وضع قيد فريد (Unique Constraint) مركب على جدول الحجوزات يجمع بين معرف الرحلة ورقم المقعد:
```sql
UNIQUE("flightId", "seatNumber")
```
هذا القيد يمنع فيزيائياً إدخال سطرين في جدول الحجوزات يحملان نفس الرحلة ونفس المقعد، مما يضمن أمان البيانات حتى لو حدث فشل في كود التطبيق.

### ثانياً: مستوى التطبيق وقفل التزامن (Pessimistic Locking)
لمنع حدوث أخطاء استثناءات (Exceptions) عشوائية من قاعدة البيانات عند محاولة الحجز المتزامن، نقوم بقفل الصف الخاص بالرحلة بشكل تشاؤمي عند القراءة:
```typescript
const flight = await entityManager.findOne(Flight, flightId, {
  lock: { mode: 'pessimistic_write' },
});
```
* **طريقة العمل**: عندما يبدأ المستخدم (أ) عملية الحجز، يقوم أمر `pessimistic_write` (المترجم في SQL لـ `SELECT ... FOR UPDATE`) بقفل صف الرحلة في قاعدة البيانات.
* إذا حاول المستخدم (ب) حجز مقعد على نفس الرحلة في نفس الميكروثانية، سيجبره النظام على الانتظار حتى تنتهي عملية المستخدم (أ) تماماً (Commit أو Rollback).
* بعد انتهاء حجز المستخدم (أ)، تبدأ عملية المستخدم (ب) فتجد أن مقعد المستخدم (أ) قد تم حجزه بالفعل وعدد المقاعد الشاغرة قد قل، فيعطيه التطبيق رسالة خطأ واضحة ومنظمة (`Seat is already taken`) بدلاً من حدوث تعارض أو حجز مزدوج.
